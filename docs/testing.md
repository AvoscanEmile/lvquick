# The `lvq` Quality Assurance & Verification Manual

This document provides a deep-dive into the multi-layered testing harness that guards the integrity of the `lvq` state engine. Because `lvq` operates as a privileged service performing destructive block-storage operations, we maintain a "Quality-First" architecture where every system invariant is backed by mathematical or empirical proof.

## Domain: Core Arithmetic & Data Integrity

**Primary Location:** `src/core/kani_proofs.rs`, `src/core/tests.rs`

This layer focuses on the "small math" that powers the entire system. Because a single overflow in byte calculation can lead to catastrophic disk partitioning errors, we treat this layer with absolute mathematical rigor.

* **Formal Property: Arithmetic Safety**
* **Checked via:** Kani Formal Proofs.
* **Invariant:** For all symbolic `u64` inputs, `SizeUnit::to_bytes()` must result in a valid `u128` without integer overflow. Kani proves this across the entire input space, ensuring that even Exabyte-scale units—which approach the limits of 64-bit addressability—cannot crash the planner or return corrupted values.

* **Property: Parser/Serializer Reflexivity**
* **Checked via:** `proptest`.
* **Invariant:** For any generated `SizeUnit` or `LvRequest`, the identity function `FromStr(Display(obj)) == obj` must hold true. This ensures that the internal memory model used for verification is always perfectly synchronized with the CLI strings presented to the user.

* **Property: Structural Integrity**
* **Checked via:** Unit tests and `ValidPercentage` constructor.
* **Invariant:** The `ValidPercentage` type acts as a refined type that strictly rejects any values outside the `1..=100` range at the moment of creation. This prevents "impossible" percentage allocations (like 0% or 150%) from ever reaching the logic gates of the planner.

## Domain: CLI & Grammar Resilience

**Primary Location:** `src/parser/tests.rs`

The parser is the "Entry Gate." It must be resilient against both malicious input and accidental user typos to prevent garbage data from entering the planning phase.

* **Property: Chaos Resilience**
* **Checked via:** Randomized argument vector injection (`test_parse_provision_chaos_resilience`).
* **Invariant:** The parser must never panic, regardless of the "junk" strings provided. It must either return a valid `Action` or a clean `Err(String)`. This guards against `panic!` calls in the standard library or indexing errors.

* **Property: Flag Positional Independence**
* **Checked via:** Proptest `Index` picker (`test_parse_floating_auto_confirm_dynamic`).
* **Invariant:** Global flags (like `-y` / `--auto-confirm`) must be correctly identified regardless of their position in the argument vector. The test dynamically injects flags into random valid indices of a generated command to ensure the state machine doesn't lose track of its position or drop subsequent arguments.

## Domain: Temporal Planning & Instruction Ordering

**Primary Location:** `src/planner/tests.rs`

LVM is an imperative system where the order of operations is critical. The planner tests enforce the "LVM Law" on every generated `Draft`.

* **Property: Object Lifecycle Invariants**
* **Checked via:** Sequential index scanning (`test_plan_provision_ordering_and_consistency`).
* **Invariant:** A `Call::VgCreate` must appear in the plan before any `Call::LvCreate` referencing that VG. Similarly, `Mkdir` must precede `Mount`. These tests ensure that the imperative sequence is always logically sound.


* **Property: Device Path Propagation**
* **Checked via:** Reflexive path matching.
* **Invariant:** The device path generated by `LvCreate` (e.g., `/dev/vg_name/lv_name`) must be the exact same path consumed by subsequent `Mkfs`, `Mount`, and `Fstab` calls. Any deviation results in a "Dirty" system state failure.

* **Property: Snowflake Logic (Swap)**
* **Checked via:** Targeted unit tests (`test_swap_special_handling`).
* **Invariant:** If an LV is marked as `Swap`, the planner must generate `MkSwap` and a `none` fstab entry, specifically omitting `Mkfs` or standard `Mount` calls. This ensures domain-specific logic is correctly applied to special filesystem types.

## Domain: System-State Simulation (The Oracle)

**Primary Location:** `src/verifier/tests.rs`

The verifier is tested against a "Mock World." We generate thousands of randomized `SystemState` scenarios to ensure the verifier makes the right decision in every possible universe.

* **Property: Idempotency World-Match**
* **Checked via:** Three-World Strategy (`Clean`, `Done`, `Dirty`).
* **Invariant:** If the `SystemState` exactly matches the `Draft`, the status must be `Done`. If it's a partial match, it must be `Dirty`. If empty, `Clean`. This is the core logic that prevents `lvq` from repeating work or "guessing" how to fix a partially broken setup.

* **Property: Hardware Failure Detection**
* **Checked via:** State Sabotage (`test_calculate_capacity_hardware_failure`).
* **Invariant:** If a path required by the `Draft` (a Physical Volume) is missing from the `SystemState.paths_exist` set, the verifier must halt the transaction with a hardware-specific error.

* **Property: The "Fstab Collision" Safety Gate**
* **Checked via:** Forbidden State generation (`test_verify_safety_logic`).
* **Invariant:** If a targeted PV contains an existing filesystem and is referenced in the host's `/etc/fstab`, the verifier must force a `Dirty` exit. This prevents `lvq` from accidentally wiping a disk that is currently active or critical to the host's boot process.

## Domain: Shell-Translation & Atomic Execution

**Primary Location:** `src/exec/tests.rs`

This layer ensures that the execution engine correctly turns abstract calls into safe, defensive shell commands.

* **Property: The Fstab-Backup Pattern**
* **Checked via:** Instruction sequence assertions (`test_fstab_lifecycle_invariants`).
* **Invariant:** Any plan containing an `Fstab` call must begin with a backup command (`cp -p /etc/fstab /etc/fstab.bak`) and end with `systemctl daemon-reload`.

* **Property: Atomic Swap Activation**
* **Checked via:** Pairwise matching (`test_swap_atomicity_invariants`).
* **Invariant:** Every `mkswap` command must be immediately followed by a `swapon` command for the exact same device.

* **Property: Precision Preservation**
* **Checked via:** Byte-flag verification (`test_vg_create_precision_invariants`).
* **Invariant:** `VgCreate` calls must use the `-s {N}B` flag, where `N` is the exact byte value. This prevents LVM from using its own default PE sizes which might conflict with the `lvq` capacity planner.

## Domain: Global Pipeline Resilience

**Primary Location:** `fuzz/fuzz_targets/full_pipeline.rs`

The "Stress Test" layer targets the entire gauntlet from `parser` to `exec`.

* **Property: Non-Crashability**
* **Checked via:** `cargo-fuzz`.
* **Invariant:** No arbitrary byte stream should ever cause a panic or hang at any stage of the pipeline. If the fuzzer finds an input that parses but causes a panic in the planner or verifier, it is considered a critical security/stability bug.

## Domain: Reflexive VM Real-World Testing (v0.2.0)

**Primary Location:** `tests/e2e_reflexive/`

The final validation layer bridges the gap between simulation and reality using ephemeral virtual machines.

* **Property: State Symmetry**
* **Checked via:** VM loopback testing.
* **Invariant:** After executing a `provision` command followed immediately by a `decommission` command, the VM's storage metadata, mount tables, and fstab must be identical to the pre-provision state. This proves that the project can not only build storage but also "leave no trace" when requested.

### CI/CD Integration & Release Gates

The testing harness is enforced via two primary GitHub Action workflows:

* **`ci.yml` (Pre-Merge):** Runs **10,000** proptests per suite, executes all Kani proofs, and runs a 60-second fuzzing smoke test.
* **`cd.yml` (Post-Tag):** Scales proptests to **100,000** per suite and runs a **600-second** extended fuzzer. **Publication of the release binary is strictly gated behind these results.** If a single property fails at 100,000 iterations, the release is aborted.
