# Devlog Entry 05 — Orchestrating Deterministic Verification and the Provision Pipeline

**Date:** 2026-03-05  
**Author:** Emile Avoscan  
**Target Version:** 0.1.0  

## Main Objective

The primary focus of this comprehensive development cycle was twofold: the architectural finalization of the Verifier module and the complete orchestration of the `lvq provision` command pipeline. Initial efforts were directed toward managing complex system states through simulation, but the objective evolved into a pursuit of absolute deterministic reliability.

The ultimate goal was to establish a strict, state-machine-driven pipeline where hardware intent is parsed, mathematically verified, authorized by the user, and finally executed. This required replacing heuristic-based "cleverness" with a rigid, two-pass verification model that ensures storage integrity by explicitly rejecting "weird middle" states, backed by a post-execution convergence check to guarantee the physical reality of the Linux storage stack perfectly matches the parsed draft.

## Implementation

### Pipeline Orchestration and State Management

The core orchestration logic was established within the `main.rs` file, implementing a strict, sequential pipeline to transition the system through varying states of trust.

* **Privilege Gates:** An initial administrative privilege check (`is_root`) was enforced to prevent unauthorized parsing or execution.
* **Lifecycle Definition:** An Action is generated by the parser, converted into a "Pending Draft" by the planner, advanced to a "Ready Draft" by the verifier, and then translated into an "Unauthorized Exec" instruction set.
* **Execution & Convergence:** A user confirmation gate must be passed before promoting instructions to an "Authorized Exec" state. Finally, a post-execution convergence check invokes the verifier a second time to guarantee the system state matches the `Done` status, exiting with specific failure codes if the hardware does not reflect the executed commands.

### Shift from Virtual Simulation to Binary Invariants

Initially, a `ProvisionState` struct was proposed as a "virtual canvas" for simulating pending changes. However, simulating every potential partial state created a "state explosion" that introduced brittleness. The architecture pivoted toward a Binary Verification Counter model. The system is now treated as a circuit that is strictly:

* **"Open"** (Clean)
* **"Closed"** (Done)
* **"Broken"** (Dirty)

This decision perfectly aligns the Verifier with the project's core pillar of "Refusal over Confusion."

### Designing the Two-Pass Verification Pipeline

The verification logic was decomposed into two distinct, non-nested functions orchestrated by `main.rs`:

1. **`verify_done` (The Existence Pass):** A loop iterates through the `Vec<Call>` and increments a counter for every action that already matches the current system state. This identifies if the project is already complete or if a "Dirty" state exists (only some components are present).
2. **`verify_possible` (The Feasibility Pass):** Running only if a `Clean` status is confirmed, this pass handles the "LVM Math," including Physical Extent (PE) rounding and metadata overhead calculations, ensuring the hardware can physically sustain the intended plan.

### Refinement of the Intermediate Representation (IR)

The `Draft` struct was updated to include an `auto_confirm` flag and a stateful `DraftStatus` enum. By embedding the status directly into the IR, the `Draft` transforms into a self-documenting contract. The lifecycle is defined as: `Pending` → (`Clean` → `Ready`) OR `Done` OR `Dirty` OR `Invalid`. This allows the `Draft` to carry its own validation history throughout the pipeline, from Planner to Executor.

### Parser Integration and Argument Deconstruction

The `parser/mod.rs` and `parser/provision.rs` modules were finalized to interpret complex command-line arguments. Logic was implemented to:

* Handle colon-separated values for multiple Physical Volumes (PVs).
* Extract custom Physical Extent (PE) sizes from the Volume Group (VG) string.
* Utilize the `LvRequest` structure to parse multi-segment logical volume definitions, specifically mapping sizes, filesystem types, and optional mountpoints into the `Command::Provision` enum.

### Executor Refinement and Abstraction

The executor module (`exec/provision.rs`) was heavily refactored to act purely as a translator between the verified `Draft` and raw shell strings. Heavy OS-level imports (`std::fs::OpenOptions`, `std::process::Command`, `std::io::Write`) were stripped from this submodule. Actual system-level execution and logging responsibilities were delegated to the parent `mod.rs` file, ensuring command generation remains entirely decoupled from IO operations.

### Module Scoping, Visibility, and YAGNI Pruning

* **Visibility Corrections:** During Planner and Verifier implementation, visibility errors regarding interaction between `mod.rs` and child worker files were resolved. Functions were refactored from private to `pub(super)` or `pub(crate)` to allow parent modules to orchestrate logic while maintaining strict encapsulation. Scoping was cleaned up using explicit `use self::...` declarations.
* **YAGNI Pruning:** Initially, unsupported `Call` variants (`PartitionDisk` and `WipeSignatures`) were handled with explicit `Err` returns to satisfy compiler exhaustiveness without using dangerous `todo!()` panics. Adhering to the "You Aren't Gonna Need It" principle, these variants were entirely removed from the `core::mod::Call` enum, eliminating dead code warnings and ensuring future features are rigorously driven by the compiler.

## Challenges & Resolutions

* **State Explosion in Partial Recovery:** Attempting to reconcile a "partially finished" state (e.g., PVs created but VG missing) led to an infinite number of edge cases.
* *Resolution:* The "Idempotency for Partial States" path was abandoned in favor of Clean State Enforcement. Any state not 100% clean or 100% finished is classified as "Dirty" (Exit 4), forcing human intervention and ensuring 100% reliability over 99% "cleverness."


* **LVM Arithmetic and Metadata Overhead:** Calculating if multiple Logical Volumes would fit into a set of Physical Volumes was complicated by hidden LVM metadata and PE rounding.
* *Resolution:* The math was consolidated into `verify_possible` using a strict formula: $\text{Total Usable} = \lfloor(\text{Raw Bytes} - \text{Metadata}) / \text{PE Size}\rfloor$. Each LV is rounded up to the nearest extent before summation.


* **Architectural Fragmentation:** Early designs risked "ghost math" where the Planner and Verifier might disagree on system capability.
* *Resolution:* Clear separation of concerns: The Planner handles *Internal Integrity* (is the plan logical?), while the Verifier handles *External Capability* (does the hardware allow it?).


* **Module Pathing Restrictions:** Visibility boundaries caused initial compilation failures (Errors E0425 and E0603), isolating `parse_provision` and `plan_provision` from the main flow.
* *Resolution:* Visibility modifiers (`pub fn`, `pub mod`) and explicit scope imports (`pub use`) were applied across the hierarchy, flattening access paths.


* **Exhaustive Pattern Matching Blocks:** The Rust compiler enforced exhaustive matching (Error E0004) on the `Call` enum in the executor, blocking compilation due to unhandled hypothetical commands.
* *Resolution:* Unused variants were permanently excised rather than bypassed with wildcard matches (`_`) or runtime panics.


* **Residual Import Noise:** Residual imports triggered dead-code analysis warnings during the final build.
* *Resolution:* The codebase was audited, pruning unused components (`SizeUnit` and `PathBuf` in the verifier; IO utilities in the executor) to maintain a strict "zero warnings" standard.

## Testing & Validation

Validation was conducted iteratively using synthesized kernel loop devices to simulate physical block storage without risking host data corruption.

1. **Isolated Volume Provisioning:** A 2GB loop device (`/dev/loop0`) was provisioned with a single Volume Group and a 500MB ext4 Logical Volume. The execution plan was successfully printed, authorized, and logged to `/var/log/lvq`. The post-execution verifier confirmed a SUCCESS convergence, and `lsblk` reflected the exact hierarchy requested.
2. **Multi-Volume and Filesystem Diversity Stress Test:** The pipeline was challenged with a complex layout on a recreated 2GB loop device: a 200MB ext4 root, a 100MB swap, a 1GB xfs home, and a `100%FREE` storage volume. The executor correctly differentiated between `mkfs` and `mkswap` binaries. The `100%FREE` mathematical calculation successfully negotiated LVM metadata overhead, exactly allocating 720MB to the final volume.
3. **Spanning, Alignment, and Mountpoint Verification (Grand Finale):** Two 1GB loop devices (`/dev/loop0`, `/dev/loop1`) were passed via colon-separated syntax to create a spanned Volume Group with a strictly enforced 16MB PE size. Three volumes were requested with directory paths.
* *Observation:* The system successfully pooled the devices. The verifier accurately calculated 16MB extent rounding (e.g., 300MB to 304MB, 200MB to 208MB). The full lifecycle—`pvcreate`, `vgcreate`, `lvcreate`, `mkfs`, `mkdir -p`, and `mount`—executed sequentially. Validation via `df -h` confirmed all volumes were active, formatted, and mounted accurately across the combined 2GB pool.

## Outcomes

The direct result of this cycle is a finalized, robust verification architecture resulting in a fully functional, zero-warning binary. The `provision` command successfully orchestrates multi-device spanning, enforces strict physical extent math, differentiates between diverse filesystem initializations, and seamlessly handles directory creation and mounting.

A forensic, append-only audit log records every intent and outcome alongside a terminal-based WYSIWYG execution prompt. The `Draft` object now serves as a transactional boundary, ensuring no execution can occur unless the system is proven to be a "Blank Canvas" and the hardware has the capacity. The "Weird Middle" failure mode has been categorized as a "Drift Detected" error, fulfilling the exit code discipline defined in `architecture.md`.

## Reflection

This development cycle served as a critical reminder that in systems programming, the pursuit of a "smart" tool can often lead to a dangerous tool. By spending a full day of cognitive effort to simplify the architecture, a complex web of heuristics was collapsed into a series of predictable, binary invariants. This "distillation" process is what transformed `lvquick` from a collection of scripts into a true safety engine.

The move to treat a "Dirty" state as a hard failure acknowledges the "Boring Reliability" philosophy: the safest action a storage tool can take in an undefined state is to do nothing at all. Furthermore, isolating domain logic from system IO circumvented the inherent brittleness of traditional Bash scripting. The zero-cost abstractions of the Rust type system ensured that the rigorous validation logic incurred virtually no performance penalty, executing complex capacity and alignment calculations instantaneously.

By prioritizing total transparency—surfacing exact command strings and verifying their effects directly against the kernel (`/proc/mounts`, LVM metadata)—the utility is elevated to a predictable, production-grade orchestrator. It proves that combining Unix philosophy with strict memory and type safety yields systems tooling that feels native, robust, and mathematically sound.
